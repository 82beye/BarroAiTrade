# 호가창 초단타 스캘핑 전략 설계 (ob_scalp)

**작성일**: 2026-05-30
**목적**: 키움 REST 호가(L2 orderbook) 마이크로구조 기반 초단타 스캘핑 전략 신규.
**구현**: `backend/core/strategy/ob_scalp.py` + `backend/core/gateway/kiwoom_native_orderbook.py`

---

## 0. ⚠️ 먼저 — 전문 트레이더의 정직한 한계 고지

1. **백테스트 불가**: L2 호가(잔량 depth) 이력이 시스템에 없다(OHLCV 캐시는 가격/거래량만). 따라서 본 전략은 OHLCV 전략들과 달리 **과거 데이터로 승률/기대값을 검증할 수 없다.** 검증은 **페이퍼/shadow 실시간**으로만 가능.
2. **지연(latency) 현실**: REST 폴링은 왕복 수백 ms. 호가 마이크로구조 엣지(OFI 등)는 초~밀리초 단위로 소멸한다. 저지연 WebSocket + 빠른 체결이 없는 retail REST 환경에서는 **엣지 대부분이 소실**된다. → 본 구현은 "단타 보조 시그널 + 페이퍼 검증 토대"이지 즉시 수익 HFT가 아니다.
3. **승률 ≠ 수익**: 앞선 승률 최적화 결론(`2026-05-30-winrate-optimization.md`)대로, 타이트 TP/SL 스캘핑은 승률은 높일 수 있으나 비용(왕복 수수료 0.03% + 세금 0.18% + 슬리피지)이 얇은 엣지를 잠식한다. **거래당 기대값이 비용을 넘는지**가 페이퍼 검증의 핵심 질문.

→ 본 설계는 **"올바르게 구현된 토대"**를 제공하며, 실거래 투입은 **페이퍼 검증으로 기대값>비용 확인 후** 단계적으로만.

## 1. 데이터 — 키움 REST 호가 (ka10004)
- 엔드포인트: `POST {base}/api/dostk/mrkcond`, `api-id=ka10004` (주식호가요청).
- 응답: `sel_{1..10}th_pre_bid/req`(매도 10단계 가격/잔량), `buy_{1..10}th_pre_bid/req`(매수), `bid_req_base_tm`(시각).
- `KiwoomNativeOrderbookFetcher.fetch_orderbook(symbol) -> OrderBook` — **실거동 검증됨**(005930: 9단계 L2, best bid/ask + 잔량 수신).

## 2. 마이크로구조 신호 (순수 함수, 테스트됨)
| 신호 | 정의 | 해석 |
|---|---|---|
| **OFI(불균형)** | 상위 N단계 (Σ매수잔량−Σ매도잔량)/(합) ∈ [−1,+1] | +면 매수우위(상승압력) |
| **스프레드(틱)** | (best_ask−best_bid)/호가단위 | 좁을수록 스캘핑 R:R 양호 |
| **마이크로프라이스** | (best_bid·ask_qty + best_ask·bid_qty)/(bid_qty+ask_qty) | 큰 잔량 반대쪽으로 끌림 → 단기 방향. >mid=상방 |
| **깊이(depth)** | 상위 N단계 양측 잔량 min | 진입·청산 유동성 병목 |
| KRX 호가단위 | 2023 개편 밴드(가격대별 1~1000) | 틱 기반 TP/SL 정확도 |

## 3. 진입/청산 규칙
**진입(롱)** — 모두 충족:
1. 스프레드 ≤ `max_spread_ticks`(기본 2) — 좁은 시장만
2. OFI ≥ `imb_threshold`(기본 0.55) — 매수우위
3. 마이크로프라이스 > mid — 상방 압력 실재
4. 깊이 ≥ `min_depth`(기본 100주) — 청산 유동성
→ best_ask 즉시 진입 가정. 점수 = OFI·마이크로 강도 → 0~10.

**청산(스캘핑)** — `exit_plan`:
- TP = 진입 + `tp_ticks`(기본 3)×틱, SL = −`sl_ticks`(기본 2)×틱, breakeven = +1틱, time_exit 15:10.
- 틱 기반이라 가격대 무관 일관(KRX 호가단위 반영).

## 4. 통합 (AnalysisContext.orderbook — 첫 소비자)
- 본 전략이 `AnalysisContext.orderbook`(기존 미사용 필드)의 **첫 소비 전략**.
- 운영 배선(다음 단계, hot-path·HITL): 스캔 루프에서 `fetch_orderbook(symbol)` → `ctx.orderbook` 주입 → `OBScalpStrategy.analyze(ctx)`. 폴링 주기는 지연 한계상 별도 고속 루프 권장(분봉 데몬과 분리).
- `_DEFAULT_ENABLED`(signal_scanner) 등록은 페이퍼 검증 후.

## 5. 검증 계획 (백테스트 불가 → 페이퍼)
1. **실시간 신호 로깅(shadow)**: 장중 고속 폴링으로 ob_scalp 신호 발생 시각·OFI·스프레드·이후 N초 가격을 기록 → "신호 후 실제 단기 방향"의 적중률·기대값 측정(주문 없이).
2. **페이퍼 체결**: dry-run 으로 진입/청산 모사, 슬리피지·지연 가정 포함한 기대값 산출.
3. **기대값 > 왕복비용(0.21%+슬리피지)** 확인 시에만 소액 실거래. 승률만으로 판단 금지.
