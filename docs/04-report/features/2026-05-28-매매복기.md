## 2. 과제 1 — swing_38 강제청산 인시던트 복기

### 2-1. 증상
broker에는 swing_38 4종목(001820/006660/012330/034220)이 보유 중인데, `data/active_positions.json`은 **`{}` (실측 2 bytes, 5/29 01:27)**. 시스템 저널상 보유가 0으로 기록되어 정상 청산 경로가 작동하지 않음 → 5/29 수동/강제 청산 필요 상태. (`_backup_20260528/active_positions.json`조차 `{}` 2 bytes로 **백업으로도 복원 불가**.)

### 2-2. 근본 원인 (구조적으로 입증된 메커니즘)
`backend/core/journal/active_positions.py`의 영속화가 비원자적이고 방어 로직이 손실을 *조장*한다 — 세 결함의 결합:

| # | 결함 | 위치 |
|---|---|---|
| 1 | `save_all()`이 atomic write가 아닌 plain `self.path.write_text(...)` → write 중단 시 파일 truncate/부분손상 | `active_positions.py` save_all (≈162–167) |
| 2 | `load_all()`이 `(json.JSONDecodeError, OSError)`를 잡아 **조용히 `{}` 반환** (raise/alert/backup 없음) | `active_positions.py` load_all (≈131–134) |
| 3 | `upsert`/`remove`가 `load_all()` 결과 위에 `save_all()` 덮어쓰기 → load_all이 한 번 `{}`를 반환하면 **그 시점 살아있던 전 키 영구 소실** | `active_positions.py` (≈169–177) |

> **정직성 단서(검증자 지적)**: 위 메커니즘은 코드상 "손실이 가능한 구조"임이 입증됐다. 다만 "실제로 그 경로로 손상이 발생했다"는 런타임 증거(손상 파일·`JSONDecodeError` 로그)는 레포에 없고, 진단 문서 `2026-05-29-sync-loss-diagnosis.md` §4도 트리거를 가설 A/B/C로 표기한다. → **실제 트리거 확정은 운영 머신 로그 확인 필요**(§2-5).
>
> **전략 귀속 단서**: 4종목이 *swing_38로* 매수됐다는 점은 구버전 `order_audit.csv`에 `strategy_id`가 없어 CSV로 확정 불가하며 모니터 docstring·정황에 의존한다. broker 보유(net 양수) 사실 자체는 입증됨.

### 2-3. "예외 케이스 이전 설정 누락"의 실체
`active_positions.json={}`가 되면 `evaluate_holdings`는 broker holdings로 청산 평가는 돌리지만(`evaluate_holdings.py:109`), `contexts`를 `active_positions.get(symbol)=None`으로 채우지 못한다(`92–107`). 결과적으로 `resolve_policy`가 swing_38 키를 못 받아 **base ExitPolicy(TP+5/SL-4)로 회귀** — swing_38 의도(SL-15/TP+50/max_hold 20, `holding_evaluator.py:128–145`)와 전혀 다른 기준으로 잘못 평가된다. 또한 swing_38 비활성(`_DEFAULT_ENABLED swing_38=False`)은 **신규 진입(scan)만 차단**하며 보유분 청산 예외 로직은 0건(scanner grep 결과 liquidate/force/청산 0) → 비활성 전환이 잔여 보유분을 방치.

### 2-4. fix 상태 (★ 핵심: 예방 미적용)
진단 문서 §6의 8개 권고 중 **탐지·메타데이터(#5/#6)만 적용, 예방(#1–4)은 전부 미적용**:

| 권고 | 내용 | 상태 | 위치 |
|---|---|---|---|
| #1 | `_audit()` OSError → dead-letter + 알림 | ❌ not_applied | `live_order_gate.py` (≈236–238, logger.error만) |

| #3 | `load_all()` graceful 처리 (`{}` 조용한 반환 금지) | ✅ **적용** (동 branch) | `active_positions.py` `_recover_corrupt`/`_load_latest_backup` |

> **결론(갱신)**: 5/29 본 복기에서 **제안2(권고 #2/#3/#4)를 구현 완료** — `active_positions.py`에 atomic write + 손상 시 백업 복원(조용한 `{}` 반환 폐지) + timestamped 자동 백업 추가. 회귀 테스트 `backend/tests/journal/test_active_positions.py` 11건 통과(캐스케이드 키 소실 차단 검증 포함). 남은 예방 권고는 **#1(`live_order_gate` dead-letter)** 1건. branch `BAR-OPS-09-active-positions-atomic-write` (미커밋/미머지).

### 2-5. 재발 방지 (우선순위)
1. **`save_all()` atomic write** — `tmp.write_text(...)` → `os.replace(tmp, path)`. write 중단에도 원본 불변. (#2)
2. **`load_all()` 강화** — 파싱 실패 시 `{}` 금지 → 손상본 `.corrupt-<ts>` 대피 + 최근 백업 복원 시도, 실패 시 HIGH alert/raise. (#3) ※ #2와 **반드시 묶어 적용** (atomic write만 하면 빈 `{}`를 안전하게 덮어써 손실 지속).
3. **save_all 직전 timestamped 백업** (`data/_active_positions_history/<ts>.json`, 7일). (#4)
4. **`_audit()` OSError → dead-letter(`data/order_audit_failed.csv`) + 텔레그램 알림.** (#1)
5. **broker 잔고를 단일 진실 source로 승격**, active_positions는 보조로 강등 — 빈 dict인데 broker holdings 존재 시 경고 후 **reconcile** 경로 추가.
6. **비활성 multi-day(swing_38) 보유분 전용 청산 경로** 명시 — 비활성=신규진입 차단과 청산을 분리.
7. active_positions **파일크기(2byte=`{}`) 기반 직접 alert**로 #8 partial → full.

### 4-3. 튜닝 제안 (미적용 — HITL 승인 필요)
**제안 1 (★ 최우선, 위험 낮음): gold_zone 운영-시뮬 일관화**
